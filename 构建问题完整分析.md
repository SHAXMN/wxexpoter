# WechatExporter GitHub Actions 构建问题完整分析

## 问题时间线

### 第1轮错误: C++ 标准不兼容
```
error: "C++ versions less than C++17 are not supported."
```
**根本原因**: protobuf 34.1 依赖 abseil 20260107.1,后者强制要求 C++17+  
**解决方案**: 升级 `CLANG_CXX_LANGUAGE_STANDARD` 从 `gnu++14` 到 `gnu++17`

### 第2轮错误: silk 库缺失
```
fatal error: 'silk/SKP_Silk_SDK_API.h' file not found
```
**根本原因**: silk 是微信语音编解码库,需要从源码编译  
**解决方案**: 从 https://github.com/collects/silk 克隆并编译

### 第3轮错误: silk 编译失败
```
clang: error: unknown argument: '-enable-threads'
```
**根本原因**: silk 的 Makefile 使用 gcc 特定参数,macOS 使用 clang  
**解决方案**: 使用 `sed` 移除 `-enable-threads` 参数

### 第4轮错误: 目录不存在
```
cp: directory /usr/local/lib does not exist
```
**根本原因**: GitHub Actions macOS runner 默认不存在 `/usr/local/lib`  
**解决方案**: 在复制前创建目录

## 预期的构建流程

### 1. 安装 Homebrew 依赖 (约 2-3 分钟)
```bash
brew install jsoncpp protobuf abseil libplist opencore-amr lame
```

### 2. 编译 silk 库 (约 1-2 分钟)
```bash
cd /tmp
git clone https://github.com/collects/silk.git
cd silk
sed -i '' 's/-enable-threads//g' Makefile
make lib  # 会有很多警告,但可以忽略
```

### 3. 安装 silk 库
```bash
sudo mkdir -p /usr/local/include/silk
sudo mkdir -p /usr/local/lib
sudo cp interface/*.h /usr/local/include/silk/
sudo cp libSKP_SILK_SDK.a /usr/local/lib/
```

### 4. 构建 WechatExporter (约 3-5 分钟)
```bash
xcodebuild clean build \
  -project WechatExporter.xcodeproj \
  -scheme WechatExporter \
  -configuration Release \
  -derivedDataPath ./build \
  HEADER_SEARCH_PATHS="..." \
  LIBRARY_SEARCH_PATHS="..."
```

## 可能的后续问题

### 问题 A: lame 库链接错误
**症状**: `ld: library not found for -llame`  
**原因**: `OTHER_LDFLAGS` 中缺少 `-llame`  
**解决**: 需要修改 Xcode 项目配置添加 `-llame`

### 问题 B: 头文件路径问题
**症状**: 找不到某些头文件  
**原因**: `HEADER_SEARCH_PATHS` 不完整  
**解决**: 确保包含所有 Homebrew 库的 include 路径

### 问题 C: 架构不匹配
**症状**: `ld: warning: ignoring file ... building for macOS-arm64 but attempting to link with file built for macOS-x86_64`  
**原因**: 依赖库架构与目标架构不匹配  
**解决**: 确保所有依赖都是 arm64 架构 (GitHub Actions 使用 arm64 runner)

### 问题 D: 代码签名问题
**症状**: `errSecInternalComponent` 或签名失败  
**原因**: GitHub Actions 环境没有有效的签名证书  
**解决**: 可能需要禁用代码签名或使用 ad-hoc 签名

## 经验教训

1. **不要假设环境**: GitHub Actions runner 的环境可能与本地不同
2. **检查依赖链**: 一个库的升级可能影响整个依赖树
3. **验证每一步**: 编译成功不等于安装成功
4. **保留警告信息**: silk 的警告虽然多,但不影响功能
5. **系统性思考**: 不要只修复表面问题,要理解根本原因
